CLEAN IMPLEMENTATION PROMPT (for coding agent)

GOAL
Create a dedicated startup/update geode that seeds and incrementally updates WORLD application links (shortcuts) from a built-in list. This fixes the fresh-install case where no links exist yet, and supports upgrades by adding only newly introduced links while preserving any user rearrangements.

SCOPE / NON-GOALS

* In scope: A new geode under Appl/Startup/... that owns WORLD link seeding (initial + incremental).
* Out of scope: Library/Pref/PrefAppl must NOT be used as the automatic startup/update seeder. PrefAppl remains UI/management only.
* Do not reorder, delete, or modify existing links that were introduced in earlier versions.
* Do not touch unrelated links/items in WORLD.

TERMS

* "WORLD": the top-level desktop folder tree (with subfolders like Extras, Games, etc.).
* "Link": whatever FileMgr uses for an application shortcut/link object in WORLD (GEOS link file/record).

DATA MODEL (IN-APP RECORD LIST)
Store the complete list of seedable application links as an in-app @chunkArray (resource data), NOT as external text/config files.

Define a record struct (minimum fields):

* introducedVersion: monotonically increasing integer (word) indicating the distribution version where this link first appears
* worldSubfolder: which WORLD folder receives the link (e.g., Extras, Games, ...)
* targetGeode: the geode/app filename to link to (or full relative path if required)
* displayName: the user-visible name for the link
* (optional) any tokens/monikers/WOT fields required by the link creation API to behave exactly like FileMgr-created links

Record list requirements:

* Records must be stable and deterministic.
* Records should be processed in increasing introducedVersion order (sort in build-time list or enforce at runtime).
* Multiple records may share the same introducedVersion.

PERSISTED STATE (GEOS.INI CONTRACT)
Persist progress in GEOS.INI so the seeder can be re-run safely and can decide what to add.

Define:

* Category: choose a dedicated category for this new geode (document exact name in code)
* Key: lastInstalledVersion (integer)

Default behavior:

* If the key is missing or invalid, treat as 0 (fresh install). That causes all records with introducedVersion > 0 to be considered.

INSTALL / UPDATE SEMANTICS (CORE ALGORITHM)
On each run:

1. Read lastInstalledVersion from GEOS.INI (default 0).
2. Iterate the in-app record list and consider only records where introducedVersion > lastInstalledVersion.
3. For each such record, attempt to ensure the link exists in the requested WORLD subfolder:

   * If the link already exists (same displayName and resolves to the intended target), treat as success and do nothing.
   * If it does not exist, create it.
4. Do NOT modify any record/link with introducedVersion <= lastInstalledVersion (no recreate, no rename, no reorder, no moving).
5. After processing, write back lastInstalledVersion according to the partial-success policy below.

IDEMPOTENCY

* Running the geode multiple times at the same system version must result in no changes after the first successful run.
* Creating a link must be safe against duplicates; do not create a second link if an identical one already exists.

ERROR HANDLING + PARTIAL-SUCCESS POLICY
Define behavior for these cases (must be implemented deterministically):

* Target WORLD subfolder missing:

  * Attempt to create the WORLD subfolder path if that is consistent with system conventions; otherwise skip the record and treat as failure for advancement (see policy below).
* Target geode missing/unavailable:

  * Do not create a dead/broken link.
* Link creation fails (API error):

  * Treat as failure.
* Name collision (a non-link item already exists with the desired displayName):

  * Do not overwrite or delete the existing item.

Version advancement policy (single lastInstalledVersion number):

* Advance lastInstalledVersion only up to the highest *contiguous* introducedVersion for which ALL records of each version were successfully handled (created or already present/valid).
* If any record for version V fails, do not advance beyond V-1. (This guarantees retry on next run and prevents silently skipping missing links.)
* Persist the updated lastInstalledVersion only after processing completes.

USER EXPERIENCE / LOGGING

* No console logging. If user-visible reporting is required, use minimal, non-intrusive system-appropriate notification; otherwise fail silently but do not advance lastInstalledVersion past failures.

INTEGRATION TRIGGER POINT

* Integrate this geode into the existing startup/update orchestration under Appl/Startup.
* It must run on fresh installs and on upgrades (i.e., whenever the system starts after an update).
* Do not couple it to PrefAppl UI. It should run headlessly and quickly.

ACCEPTANCE CRITERIA

1. Fresh install (no links exist):

   * After first run, WORLD contains the predefined links in the intended subfolders for all records up to current distribution version.
   * lastInstalledVersion is written and reflects the highest fully processed introducedVersion.
2. Upgrade from an older install:

   * Only links with introducedVersion > stored lastInstalledVersion are added.
   * Any existing user-rearranged older links remain untouched (no moves, no renames, no reorder, no recreate).
3. Re-run behavior:

   * Re-running with unchanged record list and same lastInstalledVersion performs no changes.
4. Failure behavior:

   * If a record for version V cannot be installed, lastInstalledVersion does not advance beyond V-1, ensuring retry later.

FILE-LEVEL IMPLEMENTATION MAP (PROPOSED)

* New geode directory: Appl/Startup/WorldSeed (or similar; follow repo naming conventions for startup geodes)
* Core files:

  * main process module: handles startup entry, calls seed routine, exits
  * record list resource: @chunkArray containing AppSeedRecord entries
  * ini helper module: read/write lastInstalledVersion
  * world/link helper module: locate/create WORLD subfolders, detect existing link, create link
* Keep PrefAppl unchanged except for any necessary explicit "out of scope" documentation/comments if appropriate (no functional coupling).
