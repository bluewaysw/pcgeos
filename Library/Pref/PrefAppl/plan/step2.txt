GOAL
Make AppList in PrefAppl show all the apps in the SP_SYS_APPLICATION/DISTAPPL folder and mark
those applications selected, that have at least one link in the SP_APPLICATION and/or
SP_TOP/DESKTOP folders and/or their subfolders.

STEP1
On starting up, scan the SP_SYS_APPLICATION/DISTAPPL data directory for Geodes of
type GEOS application. If SP_SYS_APPLICATION/DISTAPPL is not found, popup a clear error
message and exit. Applications in the folder can be identified via FEA_FILE_TYPE of GFT_EXECUTABLE and
FEA_GEODE_ATTR / GeodeAttrs / GA_APPLICATION.

Create a list of the Geode names (GEOS long file name, FileLongName, FEA_NAME) and their GeodeToken
in an LMem ChunkArray. Duplicate app tokens in DISTAPPL should never appear so we can ignore this
special case. Assume that there will be no subdirectorys in DISTAPPL or ignore them.

Once we have all applications in DISTAPPL, we need to sort them alphabetically by name via a
helper function.

STEP2
Afterwards, scan the SP_APPLICATION folder and the SP_TOP/DESKTOP folder. In these folders
and their subfolders we are looking for links to applications only. Implement directory
traversal in a helper with explicit safeguards:
* Maximum depth constant (16)
* Do not recurse into entries flagged as links (`FA_LINK`) to avoid link loops
* Skip `.` and `..`
* Gracefully continue on unreadable directories
* Keep traversal state in heap/LMem structures (avoid large stack frames)
Add comments in `Library/Pref/PrefAppl` helper code documenting these guarantees and expected
behavior when limit is hit.

For each link found, we will need to store the traversal root (SP_APPLICATION or SP_TOP) + the relative
directory under that root (no leading \, no . segments), plus the file name and the targets'
GeodeToken in a separate ChunkArray.

The GeodeToken of the application a link is pointing to is enough to identify the
application from DISTAPPL later. In pcgeos, the GeodeToken for applications are unique
across all manufacturers as long as both GT_chars and GT_manufID match.

The Links feature in pcgeos is not well documented. Check Library/Kernel/File/fileLink.asm to see
how that feature is implemented. FileAttrs in file.h defines FA_LINK. Links can be listed
and filtered with FileEnum and filtering for FA_LINK in FileAttrs (see Appl/FFinder/FFProcess.goc).
We are not talking about links related to GPC_DESKTOP_LINKS_IN_INI in this context. Also,
GeosFileType/GFT_LINK seems not to be implemented and is not what we want.

Use a strict two-call resolution path for each FA_LINK entry: call FileReadLink() to get the
resolved target path and disk. Call `FileGetPathExtAttributes(..., FEA_FILE_TYPE, ...)` and
require `GFT_EXECUTABLE`. Call `FileGetPathExtAttributes(..., FEA_GEODE_ATTR, ...)` and
require `GA_APPLICATION`. Only then call/read `FEA_TOKEN`. Store each link record as source location
(SP_APPLICATION or SP_TOP plus canonical relative path and link filename) plus resolved
target token, with optional resolved path/disk for diagnostics.

If multiple links to the same application are found, we still store all of them in a unique way in
the list.

STEP3
Go thru the list of applications from DISTAPPL and for each app, scan the
SP_APPLICATION and the SP_TOP/DESKTOP list for links to that application. If any one link is found,
stop the search and mark that app in the DISTAPPL list as selected.

The list of applications in DISTAPPL mmust be populated with the correct information. This list
can now be used to provide the monikers for our AppList UI gadget. If an app is selected in
DISTAPPL, it must also be selected in the AppList UI gadget. Refactor MSG_PREF_APPL_LIST_MONIKER_QUERY
to query the DISTAPPL list for the GEOS long file name of the selected applications and show that
in the list as the requested moniker.

NON_GOALS
In this step of the implementation, we are not interested in what happens when you change the
selection of the AppList. In the future this will remove/add links to/from the SP_APPLICATION and
SP_TOP/DESKTOP folders but we are not implementing this now. However, we must make sure that it
can be safely and easily added later.

DESIGN NOTES
Create a robust and reusable helper function for enumerating the DISTAPPL, SP_APPLICATION
and SP_TOP/DESKTOP folders which scans either for application geodes or links to
application geodes. Make sure to use LMem ChunkArray functions for memory management and don't
forget the usual GEOS drill of MemLock/MemUnlock. ChunkArrayAppend() invalidates existing
pointers into the LMem heap, so never keep element pointers across any operation that may
resize/compact (append/insert/resize). Prefer indices, or re-derive pointers each time.
Put all helper functions in their own code file so prefappl.goc can be kept small.
