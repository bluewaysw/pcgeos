GOAL
Implement end-to-end apply behavior for `MSG_PREF_APPL_LIST_APPLY` in `Library/Pref/PrefAppl`
so selecting apps creates links in `SP_APPLICATION`, and deselecting apps removes all existing links targeting those apps.
New links are created **only** in `SP_APPLICATION` (never `SP_TOP\\DESKTOP`).

Deselect cleanup walks scanned link list records and removes token-matching links from `SP_APPLICATION`,
`SP_TOP\\DESKTOP` and all its subfolders. The link list should contain all currently existing links
in these folders.

Keep behavior robust on partial failures (continue processing remaining items).

STEP: Add reusable link operation helpers in scan/helper layer

Add helper to create a link for an app record in `SP_APPLICATION` (exactly there, not in a subfolder)
using `FileCreateLink`. If a link with this name already exists, skip. No filename normalization
rules should be needed.

The disk handle needed for `FileCreateLink` will always be SP_SYS_APPLICATION.
and the path will always be “SP_SYS_APPLICATION\\DISTAPPL”.
So you can create links with:
targetDiskHandle = SP_SYS_APPLICATION (standard path handle form),
targetPath = "DISTAPPL\\<long GEOS appname aka PAAR_name>".

Add helper to delete all scanned links that match an app token by iterating `PrefApplLinkRecord`
entries and removing each source link file (`FileSetCurrentPath` + `FileDelete`).
Use existing token comparison semantics (`PrefApplTokensMatch`) and keep MemLock/MemUnlock discipline.

STEP: Implement APPLY logic itself

At the beginning of MSG_PREF_APPL_LIST_APPLY, disable (...NOT_ENABLED) the AppList, so that
no user interaction is possible while we are busy.

In `Library/Pref/PrefAppl/prefappl.goc`, implement `MSG_PREF_APPL_LIST_APPLY` as a
reconciliation pass. Read current list selection using `MSG_GEN_ITEM_GROUP_GET_NUM_SELECTIONS`
and `MSG_GEN_ITEM_GROUP_GET_MULTIPLE_SELECTIONS` and build a desired-selection map by app index.
Call `PrefApplRescanApplicationLinks()` before making changes, so operations are based on
actual current links, not cached assumptions. Loop through all app records
(`PrefApplGetApplicationCount` + `PrefApplGetApplicationRecord`):
If desired selected:
- check scanned link list for a token match.
- if none exists in `SP_APPLICATION`, create exactly one link via helper (see above). If
an app is selected in UI and currently has links only in SP_TOP\DESKTOP (none in SP_APPLICATION),
treat the desktop-only links as already satisfied (create nothing).
If desired not selected:
- delete all scanned links matching that app token.

After all create/delete attempts, call `PrefApplRescanApplicationLinks()` again, then run
the existing “mark selected from scanned links” path so `PAAR_selected` is authoritative again.

Refresh list selection display with `MSG_PREF_APPL_APPLY_SELECTED_STATE_TO_LIST`.

Re-enable the AppList at the end, so that user interaction is possible again.

Note: Continue-on-error for per-item operations; optionally set a short status message (StatusText) if
any operation failed, but do not abort the whole reconcile pass.

GENERAL CONSIDERATIONS:

C89 variable declarations at function top.
Small stack footprint (use handles/chunk arrays for variable-size buffers).
No recursion through link entries; preserve existing traversal safeguards.
Continue-on-error where possible; surface concise status text only when useful.
Keep create/delete logic in dedicated helpers in `Library/Pref/PrefAppl/prefapplScan.goc` so `prefappl.goc` stays orchestration-only.
Add comments to help future maintainers.
